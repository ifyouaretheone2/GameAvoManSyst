## Context

后端当前以 `admin` 表存储可登录账号，`POST /admin/login` 签发 JWT，`authenticate` 中间件仅校验令牌。前端 Pinia `user` 存储 token 与管理员信息；路由以 `/` 为公开首页，`/login` 为登录页，`/admin/*` 受 `requiresAuth` 保护。尚不存在自助注册、角色区分或“将普通用户提升为管理员”的接口与界面。

## Goals / Non-Goals

**Goals:**

- 支持自助注册，注册账号默认不具备后台管理能力（普通用户角色）。
- JWT 与中间件能区分管理员与普通用户；后台写操作仅管理员可访问。
- 提供受控的管理员提升能力（由已是管理员的账号发起）。
- 访问应用根路径时的默认体验为登录页（与提案一致）；登录后按角色跳转。

**Non-Goals:**

- 第三方 OAuth、邮箱验证、短信验证码、密码找回（可后续单独立项）。
- 细粒度 RBAC（多角色、权限点矩阵）。
- 将现有公开内容页从站点中移除（仅调整路由映射，内容可迁至新路径）。

## Decisions

1. **数据模型：在现有 `admin` 表增加角色字段**

   - 增加 `role`（建议枚举：`admin` | `user`），现有行一次性回填为 `admin`，新注册默认为 `user`。
   - **理由**：改动面小于新建 `users` 表并全量迁移；与当前 `adminController` / SQL 查询路径兼容。
   - **备选**：新建 `users` 表并逐步废弃 `admin` 表名——命名更清晰，但迁移与引用替换成本更高。

2. **API 形态：在现有 `admin` 路由域内扩展**

   - 新增 `POST /admin/register`（或项目统一的 `/api` 前缀下的等价路径），与 `POST /admin/login` 并列；登录查询需允许 `role` 为 `user` | `admin` 的账号均可登录（若产品希望普通用户仅浏览前台、不允许登录后台，则登录成功后由前端与路由守卫分流）。
   - **理由**：减少全局路由前缀重构；与 `adminRoutes.js` 扩展一致。
   - **备选**：引入 `/auth/*` 命名空间——更清晰，但需同步前端 `api` 基路径与文档。

3. **JWT 载荷包含 `role` 与 `id`**

   - `authenticate` 将 `req.user.role` 提供给控制器；新增 `requireAdmin` 中间件：`role === 'admin'` 才进入后台管理 API。
   - **理由**：服务端强制授权，避免仅靠前端隐藏菜单。

4. **提升管理员接口**

   - `POST /admin/users/:id/promote`（或 `PATCH /admin/users/:id` with `{ role: 'admin' }`），仅 `requireAdmin`。
   - 约束：目标用户须已存在且当前为 `user`；禁止匿名调用；可选禁止最后一个管理员被降级（若后续提供降级接口）。
   - **理由**：语义明确、易审计。

5. **默认路由**

   - 将原 `Home` 内容挂到 `/home`（或 `/site`），`/` 重定向到 `/login`；已登录且 `role === 'admin'` 访问 `/login` 时仍进入 `/admin`；普通用户访问 `/login` 登录后可进入 `/home` 浏览公开内容。
   - **理由**：满足“初始为登录页”，同时保留公开站点入口。
   - **备选**：根路径直接渲染 `Login` 组件而不改 `Home` 路径——需二选一避免与公开首页冲突。

## Risks / Trade-offs

- **[Risk] 根路径从首页改为登录/重定向导致外部收藏夹失效** → 在变更说明与 README 中标注 **BREAKING**，必要时增加短期重定向兼容或保留旧路径别名。
- **[Risk] 表名 `admin` 与“普通用户”语义混淆** → 代码与 OpenAPI 注释中使用“账号/用户”表述，后续迭代可计划表重命名。
- **[Risk] 开放注册带来垃圾账号** → Non-Goal 中已排除复杂治理；可在配置中增加 `ALLOW_REGISTRATION` 开关（可选实现）。

## Migration Plan

1. 数据库迁移脚本：添加 `role` 列 + 默认值 + 现有数据 `UPDATE` 为 `admin`。
2. 部署顺序：先迁移与后端发布（新列、新接口），再发布前端（路由与注册页），或同一版本原子发布。
3. 回滚：保留列默认值与旧前端路由包即可回退客户端；数据库列可保留忽略（不回删）以降低回滚风险。

## Open Questions

- 普通用户登录后是否允许进入 `/admin` 任意子路径（建议：**不允许**，仅 `admin` 角色进入后台布局）。
- 注册字段范围：是否必须 `nickname`、密码强度规则与用户名唯一性校验强度。
